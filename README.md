# Дивошкола на GitHub
Сайт Дивошколи — це інформаційний продукт з відкритим вихідним кодом, опублікований на GitHub під ліцензією MIT. Ми хочемо, щоб знання були доступнішими, а процеси здобування освіти та оволодіння навичками — якіснішими. Ми відкриті до співпраці і будемо вдячні за будь-яку допомогу в нашій роботі.

## Як це працює
Сайт наразі є повністю статичним і автоматично збирається за допомогою [VitePress](https://vitepress.dev/) з форматованих текстових файлів Markdown. Такий вибір дуже спрощує спільну роботу над матеріалом і його публікацію, використовуючи лише інфраструктуру GitHub. Будь-хто може клонувати матеріал, відредагувати і запропонувати зміни.

Основний репозиторій містить каркас сайту, деякі розділи і загальні сторінки, конфігурацію та збірку. Контент проєктів підключається з окремих репозиторіїв через Git submodule. Наприклад, таким чином підключено розділ [`knots/`](./knots) з репозиторію [`dyvoshkola/knots`](https://github.com/dyvoshkola/knots). Під час локальної розробки і в GitHub Actions вміст проекту завантажується разом з основним сайтом, після чого VitePress збирає все в один статичний сайт для публікації.

## Клонування та інсталяція
Для локального запуску потрібні `git`, `Node.js 20` і `npm`. Репозиторій потрібно клонувати разом із підмодулями та інсталювати залежності:

```bash
git clone --recurse-submodules https://github.com/dyvoshkola/dyvoshkola.github.io.git
cd dyvoshkola.github.io
npm install
```

Якщо потрібно встановити залежності точно у тих версіях, які вже зафіксовані в `package-lock.json`, наприклад для повторюваної збірки або перевірки роботи, замість `npm install` слід використовувати команду:

```bash
npm ci
```

Для локального запуску використовуємо:

```bash
npm run docs:dev
```

Для production-збірки:

```bash
npm run docs:build
```

Якщо репозиторій вже було клоновано без підмодулів (за посиланнями на сторінки з підмодулів відображаються помилки 404), їх можна підтягнути окремо:

```bash
git submodule update --init --recursive
```

Після оновлення основного репозиторію або підмодулів може знадобитися повторний запуск `npm install` або `npm ci`.

## Як запропонувати зміни
Якщо ви бажаєте запропонувати зміни, але не плануєте одразу чи взагалі редагувати код, скористайтесь механізмом [Issues](https://github.com/dyvoshkola/dyvoshkola.github.io/issues), що є в кожному репозиторії Дивошколи — там можна описати проблему, ідею або покращення та обговорити рішення.

Якщо ви можете самостійно відредагувати код і запропонувати оновлення, рекомендуємо зробити це через GitHub fork і pull request. В такому випадку ви працюватимете у власній копії репозиторію і зможете вільно пушити свої гілки без необхідності мати прямі права на запис в основний репозиторій.

І, звісно, ви можете приєднатись до однієї з команд проектів Дивошколи. Команди працюють з нашими репозиторіями напряму, без форків. А також мають доступ до внутрішніх робочих інструментів.

Для редагування потрібно спочатку клонувати на локальну машину саме той репозиторій, у який ви плануєте вносити зміни: основний `dyvoshkola/dyvoshkola.github.io`, репозиторій проєкту, або власний форк якогось з них. Інструкції щодо клонування див. у попередньому розділі.

Створіть окрему гілку для змін (вкажіть власну назву гілки):

```bash
git checkout -b my-branch
```

Після цього внесіть потрібні зміни у файли, додайте їх до коміту:

```bash
git add .
```

Збережіть зміни в історії git з коротким описом:

```bash
git commit -m "Короткий опис змін"
```

Відправте свою гілку у репозиторій:

```bash
git push origin my-branch
```

Після цього на GitHub відкрийте pull request з вашої гілки у `dyvoshkola/dyvoshkola.github.io` або у відповідний репозиторій проєкту, якщо зміни вносилися саме туди.

## Публікація на GitHub
Публікація на GitHub налаштована через GitHub Actions: workflow `.github/workflows/deploy.yml` на кожен push у гілку `main` ініціалізує submodule, встановлює залежності, збирає сайт і деплоїть результат у GitHub Pages.
